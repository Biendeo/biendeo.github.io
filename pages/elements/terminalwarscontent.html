Terminal Wars

Terminal Wars is a game I created for my first semester COMP1917 class at university. I made it to demonstrate my ability to plan and debug software. In retrospect, I put in a lot of work planning the project, but I think I planned it so much that I ended up not having a lot of room to debug the program (as everything was implemented quite nicely).

The game is based on Advance Wars: Dual Strike, and features most of the gameplay features of it. In the game, you create units at bases, and use them to destroy enemy units and capture buildings. Buildings earn you income each turn, which allow you to get more powerful units more often. The goal is to capture your opponent's HQ building. Different units have their strengths and weaknesses, and so figuring out which unit is right for the job is key to winning. Units also have movement speeds on different kinds of terrain, and have a limited amount of ammo and fuel on them, so they need to spend a turn sitting on a captured city to heal. On top of that, only infantry and mech units can capture buildings, but they're also the most vulnerable units. In the original game, the APC unit was a transport vehicle used to carry these two units with a bit more protection and greater movement speed. There's also naval and air units to spice things up.


So implementing this was a bit of a tricky part. Planning the whole thing required me to analyse the game's concepts and mechanics, and try and recreate them in C. The biggest challenge I had was representing and interacting with the game. I spent a bit of time looking up easy ways to do this system, and I came across this convenient header called rlutil.h, which had simple cross-platform functions for handling coloured console output and key input settings. The only cross-platform issue I had was that the arrow keys didn't work on Unix systems, so all I had to do was just redefine them to other keys. I had an issue where I couldn't include the header in multiple files, so I just tried to focus all of its commands into the DrawingUI.c file. I still had some things that carried over to MainGame.c, so I had to do some messy stuff there.

Another part of planning was figuring out how to store and handle the data. I originally was going to showcase the ADT setup that I had learned about, which is why the project is split up into multiple files. The use of an ADT is to hide the back-end info from someone developing the front-end. They both know the functions that will be called, and what they should expect them to do, but the actual implementation can vary. This allows someone to modify and fix a back-end problem without modifying what happens on the front-end. Of course, it's a bit needless if you're the only person working on something. A lot of my functions ended up directly accessing the data anyways.

But I spent a lot of time preparing how the whole thing was stored. I knew that each player had certain properties related to each one, and then there was game specific data that would also be stored. The most daunting part was figuring out how the program itself would "create" units. At first inspection, I thought it was impossible. Then I started thinking about how it could be done, and figured out that technically all the units could be loaded, but they could just be uninitialised until needed. This meant that I could just give them all values when I needed, and default them all when they were destroyed. They probably do things like this in other games too (like there'd be hard-coded limits for how many of a certain object).

One of the most annoying parts was filling in the statistics for each unit. Every unit has about ten different properties describing its movement costs, type, etc. On top of that, every unit has a specific value for the amount of damage it does to other units. A few units have two weapons, and thus have two values for some enemies. This meant that about 2,000 lines of code were just ifs and else if statements.

I gave myself plenty of time to do this project. I think I set myself about 6 weeks to do it, and I finished it in about 3 (although I spent way too much of my free time working on it). There's still a few parts that I've been a bit lazy to finish (there's no buy menu for boats and planes, so you can't buy those yet (although metadata is there, so you just need to make that interface)). You also can't have units carrying other units such as the APC (which makes it completely useless right now). Finally, there's no win-state in the game. I've been really lazy, so I never really put those in. But I also feel as if the original submission is probably the best thing to leave it with. It was ultimately a project, and I'll keep it as that. It was a good experiment, and it used all the concepts I had learned so far (although my linked list viewer was a lot more perfect in that regard).

If I could do it differently, I'd definitely clean the system a bit. After learning about enums and switches on the side, I definitely see how useful those would've been. But in the end, I had to stick to the course's style guide.
